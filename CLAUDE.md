# SMM694 RAG Prototype — Claude Code Guide

## Project overview
Retrieval-Augmented Generation (RAG) prototype built for SMM694 Applied NLP assignment.
Ingests PDF documents, embeds them into a ChromaDB vector store, retrieves relevant chunks at query time, and generates answers via an LLM. A Streamlit dashboard (`app.py`) exposes the pipeline interactively. RAGAS evaluation is in `evaluate.py`.

## Stack
- **Embeddings / vector store**: ChromaDB (`chroma_db/` — git-ignored, regenerated by `src/embed.py`)
- **LLM**: configured via `GEMINI_API_KEY` in `.env`
- **UI**: Streamlit (`app.py`)
- **Evaluation**: RAGAS (`evaluate.py`)

## Key files
| File | Purpose |
|---|---|
| `src/download.py` | Download 10-K filings from SEC EDGAR → `data/raw/` |
| `src/ingest.py` | Parse raw filings → text chunks |
| `src/embed.py` | Embed chunks → ChromaDB |
| `src/retrieve.py` | Query vector store → top-k chunks |
| `src/generate.py` | Chunks + prompt → LLM answer |
| `src/utils.py` | Shared helpers (logging, config loading) |
| `app.py` | Streamlit entry point |
| `evaluate.py` | RAGAS evaluation script |

## Running the project
```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Copy and fill in API keys (GEMINI_API_KEY, SEC_USER_NAME, SEC_USER_EMAIL)
cp .env.example .env

# 3. Download 10-K filings from SEC EDGAR
python src/download.py

# 4. Parse filings into chunks and embed into ChromaDB
python src/ingest.py
python src/embed.py

# 5. Launch the app
streamlit run app.py

# 6. Run evaluation
python evaluate.py
```

## Git-ignored vs committed data
Ignored (never committed):
- `venv/`, `.env` (secrets), `__pycache__/`, `*.pyc`
- `data/raw/` — SEC EDGAR filings (HTML/txt, ~227MB)

Committed on purpose (so a fresh clone / the deployed image has a working vector
store without re-downloading and re-embedding):
- `data/processed/chunks.json` — chunked text
- `chroma_db/` — vector store

## Data hygiene (important — this repo is public)
Because `chunks.json` and `chroma_db/` are committed to a **public** repo, per-chunk
metadata must never contain machine-local information. In particular the `source`
field must be a **repo-relative** path (e.g.
`data/raw/sec-edgar-filings/BAC/10-K/.../full-submission.txt`), never an absolute
path — an absolute path leaks the OS username, Google account email, and private
folder structure. `src/ingest.py` enforces this via `txt_path.relative_to(repo_root)`;
keep it that way, and re-run `src/embed.py` after any change to `chunks.json` so the
vector store stays in sync.
